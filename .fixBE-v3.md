# 0. Chẩn đoán ngắn: vì sao hiện tại UI vẫn “đẹp nhưng đứng yên”

Hiện tại hệ thống đang đứt ở 6 điểm chính:

1. **Load folder multicam mới chỉ tồn tại ở browser**, chưa có contract upload/staging vào backend.
2. **Load resources.txt mới dừng ở parse config**, chưa biến thành preview RTSP có thể chọn được ở `MULTICAM`.
3. **Start chưa tách rõ trạng thái `loaded` và `running`**, nên đang có hành vi “load xong là như sắp chạy”, trong khi user muốn chỉ chạy khi bấm `START`.
4. **Original / Processed snapshots chưa được quản lý như hai luồng độc lập**, nên 2 khung này không show ổn định.
5. **Registration đang lẫn 2 nguồn camera**: browser webcam preview và backend capture source.
6. **Socket event contract chưa đủ chuẩn**, nên UI không nhận đều được event, log, metrics, clip queue.

Nếu không vá 6 điểm này, bạn sẽ luôn gặp hiện tượng:
- chọn folder nhưng backend không xử lý thật,
- chọn TXT nhưng RTSP không hiện trong MULTICAM,
- bấm START mà Original/Processed không lên,
- đăng ký face có modal nhưng không hoàn tất,
- Live Preview hiện sai nơi,
- log không ra đúng bảng.

# 1. Nguyên tắc kiến trúc phải giữ

## 1.1 Không được tự ý xử lý video ngay khi user vừa load

Đúng theo yêu cầu mới:

- `Load folder` chỉ có nghĩa là **nạp danh sách clip vào trạng thái chờ**.
- `Load resources.txt` chỉ có nghĩa là **nạp danh sách camera RTSP vào selector**.
- Chỉ khi user nhấn `START` thì backend mới bắt đầu:
  - đọc clip / RTSP,
  - chạy pose/ADL,
  - cập nhật Original,
  - cập nhật Processed,
  - ghi metric log,
  - ghi kết quả vào `data/`.

## 1.2 LIVE PREVIEW không được tự bật khi chỉ load multicam

Đúng hành vi cần giữ:

- `MULTICAM` là nơi hiển thị nguồn chính khi user bật preview RTSP.
- `ORIGINAL` là nơi hiển thị frame gốc của clip đang xử lý.
- `PROCESSED` là nơi hiển thị overlay pose/ADL.
- `LIVE PREVIEW` bên phải **chỉ mở khi user bấm Preview trên clip card**.

## 1.3 Log cũ không xóa khi chuyển sang short video mới

Backend phải emit log theo kiểu append-only trong phiên chạy.

Khi chuyển clip mới:
- bắt đầu log mới cho clip đó,
- nhưng **không xóa log cũ** trong bảng runtime.

Có thể dùng chính sách giữ tối đa 300–500 dòng gần nhất.

## 1.4 Registration phải có một “source of truth” duy nhất

Khuyến nghị product:
- backend là nơi capture snapshot đăng ký (`local` hoặc `rtsp`),
- frontend chỉ là nơi mở modal, xem preview, theo dõi tiến độ.

**Không lấy browser webcam làm nguồn đăng ký thật** nếu backend registration service vẫn chạy từ OpenCV local source.

# 2. Hành vi mục tiêu sau khi vá

## 2.1 Luồng multicam folder

1. User nhấn `Load` / chọn folder `.mp4`
2. Frontend upload/stage danh sách clip sang backend
3. Backend parse + sort time-first + trả queue
4. UI hiện clip cards ở `VIDEO CLIPS`, trạng thái `ready`
5. **Chưa chạy gì cả**
6. User nhấn `START`
7. Backend lần lượt xử lý từng clip:
   - Original hiển thị clip gốc đang xử lý
   - Processed hiển thị frame overlay pose/ADL
   - bảng đen log metric liên tục
   - khi clip xong thì lưu output vào `data/output_pose/...`
   - chuyển clip kế tiếp
8. `LIVE PREVIEW` chỉ hiện khi user click `Preview` ở card clip

## 2.2 Luồng RTSP qua resources.txt

1. User nhấn `TXT` và chọn `resources.txt`
2. Backend parse file, trả danh sách camera kiểu:
   - `Cam Nha __ rtsp://...`
3. Frontend populate `slotSel1`
4. User chọn một camera
5. User nhấn `Show`
6. Backend trả snapshot camera đó qua `/api/cameras/<cam_id>/snapshot`
7. `MULTICAM` hiện preview RTSP
8. Khi user nhấn `START`, pipeline recorder/recognizer mới chạy thật

## 2.3 Luồng registration

1. User nhấn `Open`
2. UI chỉ bật/tắt preview webcam nếu bạn vẫn muốn giữ trải nghiệm cục bộ
3. User nhấn `New`
4. Modal lấy:
   - name
   - age
   - person_id tự tăng
5. Backend `/api/registration/start` khởi tạo session
6. `/api/registration/snapshot/<session_id>` trả frame có bbox + overlay text
7. socket emit `registration_progress` và `registration_done`
8. Khi done:
   - user profile được lưu,
   - recognizer refresh database,
   - UI đóng modal đúng cách

## 3.1 App Factory — tăng upload limit, giữ callback Producer → Consumer

### Mục tiêu
- upload được nhiều video demo lớn hơn 16MB,
- recorder đẩy clip sang recognizer tự động,
- Socket.IO init ổn định.

### Hiện trạng
App factory đang để `MAX_CONTENT_LENGTH = 16 * 1024 * 1024`, quá nhỏ cho video demo.

### Code mẫu đề xuất (`app/bootstrap/app_factory.py`)

```python
import os
import logging
from pathlib import Path
from flask import Flask, send_from_directory
from flask_cors import CORS
from flask_socketio import SocketIO

from app.utils.config_schema import AppConfig
from .logging_setup import setup_logging

logger = logging.getLogger("[AppFactory]")
socketio = SocketIO()


def create_app(static_dir: Path, config: AppConfig) -> Flask:
    setup_logging(config.logging)
    logger.info("Initializing CPose application factory...")

    flask_app = Flask(
        __name__,
        static_folder=str(static_dir),
        static_url_path="/static",
    )

    flask_app.config["SECRET_KEY"] = os.environ.get("SECRET_KEY", "cpose-dev-secret")

    # FIX v3: tăng upload limit cho demo video/folder
    flask_app.config["MAX_CONTENT_LENGTH"] = 512 * 1024 * 1024  # 512MB

    CORS(flask_app, resources={r"/*": {"origins": config.server.cors_origins}})

    from app.services.recorder import RecorderManager
    from app.services.recognizer.orchestrator import RecognizerService

    recognizer = RecognizerService(config, socketio)
    recognizer.start()

    recorder = RecorderManager()

    flask_app.config["recorder"] = recorder
    flask_app.config["recognizer"] = recognizer

    # Producer -> Consumer callback
    def on_clip_ready(clip_path: Path, cam_id: str):
        logger.info("Auto-processing trigger: %s from %s", clip_path.name, cam_id)
        recognizer.enqueue_clip(cam_id, Path(clip_path))

    flask_app.config["on_clip_ready_cb"] = on_clip_ready

    _register_blueprints(flask_app)
    _register_ui_routes(flask_app, static_dir)

    socketio.init_app(
        flask_app,
        cors_allowed_origins=config.server.cors_origins,
        async_mode=config.server.socket_async_mode,
    )

    _register_socket_handlers()
    return flask_app
```

### Vì sao cần sửa
Nếu không nâng upload limit:
- load multicam bằng upload sẽ fail ngầm hoặc trả 413,
- user tưởng backend hỏng,
- trong khi thực ra request bị chặn ngay từ Flask.

## 3.2 Socket.IO handlers — chuẩn hóa event contract cho dashboard

### Mục tiêu
Tạo 1 nơi duy nhất để emit các event dashboard cần nghe:
- event log mini bên trái,
- metric log đen bên dưới,
- progress pose,
- clip saved,
- registration progress/done.

### Code mẫu đề xuất (`app/api/ws_handlers.py`)

```python
from __future__ import annotations

import logging
import threading
import time
from typing import Any

from app import socketio

logger = logging.getLogger("[WS]")
_pending: dict[str, dict[str, Any]] = {}


def emit_event_log(event: str, cam: str = "--"):
    payload = {
        "time": time.strftime("%H:%M:%S"),
        "event": event,
        "cam": cam,
    }
    socketio.emit("event_log", payload)


def emit_metric_log(
    cam: str = "--",
    fps: float | int | str = "--",
    frame: int | str = "--",
    conf: float | int | str = "--",
    adl: str = "--",
    event: str = "runtime",
    mode: str | None = None,
):
    payload = {
        "time": time.strftime("%H:%M:%S"),
        "cam": cam,
        "fps": fps,
        "frame": frame,
        "conf": conf,
        "adl": adl,
        "event": event,
        "mode": mode,
    }
    socketio.emit("metric_log", payload)


def emit_pose_progress(
    clip_stem: str,
    cam_id: str,
    frame_id: int,
    total_frames: int,
    fps: float,
    conf: float,
    adl: str,
):
    pct = 0 if total_frames <= 0 else round(frame_id * 100.0 / total_frames, 1)
    payload = {
        "clip_stem": clip_stem,
        "cam_id": cam_id,
        "frame_id": frame_id,
        "total_frames": total_frames,
        "fps": fps,
        "conf": conf,
        "adl": adl,
        "pct": pct,
    }
    socketio.emit("pose_progress", payload)


def emit_clip_saved(filename: str, cam_id: str, rel_path: str, preview_path: str | None = None):
    payload = {
        "filename": filename,
        "cam_id": cam_id,
        "path": rel_path,
        "preview_path": preview_path,
        "time": time.strftime("%H:%M:%S"),
    }
    socketio.emit("clip_saved", payload)


@socketio.on("connect")
def _handle_connect():
    logger.info("UI connected via SocketIO")


@socketio.on("disconnect")
def _handle_disconnect():
    logger.info("UI disconnected via SocketIO")


def request_face_registration(clip_stem: str, cam_id: str, timeout: float | None = 300.0) -> dict | None:
    logger.info("Requesting face registration for %s (%s)", clip_stem, cam_id)
    ev = threading.Event()
    _pending[clip_stem] = {"event": ev, "payload": None}

    socketio.emit("register_face_request", {"clip_stem": clip_stem, "cam": cam_id})
    waited = ev.wait(timeout=timeout)
    entry = _pending.pop(clip_stem, None)
    if not waited or entry is None:
        logger.warning("Registration timeout for %s", clip_stem)
        return None
    return entry.get("payload")


@socketio.on("register_face_done")
def _handle_register_face_done(data):
    clip_stem = str(data.get("clip_stem", ""))
    if not clip_stem:
        return
    entry = _pending.get(clip_stem)
    if not entry:
        return
    entry["payload"] = data
    entry["event"].set()
```

### Vì sao cần sửa
Nếu không có emitter chuẩn:
- backend emit tên event lung tung,
- frontend chờ một tên khác,
- log / progress / clip queue không bao giờ lên.

```python
@api_bp.route("/pose/start", methods=["POST"])
def start_pose():
    body = request.get_json(force=True, silent=True) or {}
    mode = body.get("mode", "multicam_folder")  # rtsp | multicam_folder

    if mode == "rtsp":
        cameras_req = body.get("cameras", [])
        all_cams = _storage.parse_resources(_app_module.RESOURCES_FILE)
        selected_cams = [c for c in all_cams if c.get("cam_id") in cameras_req]
        if not selected_cams:
            return jsonify({"error": "No valid cameras for RTSP mode"}), 400

        get_recorder().start(
            cameras=selected_cams,
            storage_limit_gb=float(body.get("storage_limit_gb", _app_module.DEFAULT_STORAGE_LIMIT_GB)),
            output_dir=_app_module.RAW_VIDEOS_DIR,
            model_path=_app_module.MODEL_PHASE1,
            on_clip_ready=current_app.config.get("on_clip_ready_cb"),
        )

        _WORKSPACE.mode = "rtsp"
        _WORKSPACE.running = True
        return jsonify({"status": "started", "mode": "rtsp", "cameras": len(selected_cams)})

    elif mode == "multicam_folder":
        if not _WORKSPACE.staged_clips:
            return jsonify({"error": "No staged multicam clips. Load first."}), 400

        recognizer = get_pose()
        for clip in _WORKSPACE.staged_clips:
            cam_id = _app_module.extract_multicam_camera_id(clip) or "unknown"
            recognizer.enqueue_clip(cam_id, clip)

        _WORKSPACE.mode = "multicam_folder"
        _WORKSPACE.running = True
        return jsonify({
            "status": "started",
            "mode": "multicam_folder",
            "total_clips": len(_WORKSPACE.staged_clips),
        })

    return jsonify({"error": f"Unsupported mode: {mode}"}), 400
```

## 3.10 API config/upload + config/cameras — hỗ trợ resources.txt dạng `Cam Nha __ rtsp://...`

### Mục tiêu
Khi load file:

```text
Cam Nha __ rtsp://user:pass@192.168.100.xxx:554/stream2
```

UI phải populate lên `MULTICAM` selector, chọn được camera, và preview được snapshot.

### Yêu cầu backend
`parse_resources()` phải chấp nhận format:
- `label __ url`
- nếu không có `__` thì fallback theo format cũ

### Code mẫu logic parse

```python
def parse_resources_flexible(path: Path) -> list[dict]:
    cameras = []
    if not path.exists():
        return cameras

    with open(path, "r", encoding="utf-8") as f:
        for idx, line in enumerate(f):
            line = line.strip()
            if not line or line.startswith("#"):
                continue

            label = None
            url = None

            if "__" in line:
                left, right = line.split("__", 1)
                label = left.strip()
                url = right.strip()
            else:
                url = line.strip()
                label = f"cam{idx+1:02d}"

            cam_id = f"cam{len(cameras)+1:02d}"
            cameras.append({
                "cam_id": cam_id,
                "label": label,
                "source": url,
                "cam_url": url,
            })

    return cameras
```

### Khuyến nghị
Nếu `StorageManager.parse_resources()` đang cứng format cũ, hãy vá nó thay vì vá ở frontend.

## 3.11 Registration — backend phải là nguồn thật, UI chỉ là monitor

### Vấn đề hiện tại
Nếu frontend mở browser webcam bằng `getUserMedia()` nhưng backend registration lại capture từ OpenCV source `local`, thì 2 nguồn đó không chắc là cùng một camera.

### Cách ổn định nhất
Cho demo product:
- backend registration service là nguồn thật,
- frontend chỉ mở modal + đọc snapshot từ `/api/registration/snapshot/<session_id>`.

### Endpoint giữ nguyên nhưng contract cần chắc hơn

```python
@api_bp.route("/registration/next_id", methods=["GET"])
def get_next_registration_id():
    next_id = _registration.get_next_id()
    return jsonify({"next_id": next_id})


@api_bp.route("/registration/start", methods=["POST"])
def start_registration():
    body = request.get_json(force=True, silent=True) or {}
    source = body.get("source")
    rtsp_url = body.get("rtsp_url")
    user_name = body.get("name")
    user_age = body.get("age", "??")
    person_id = body.get("person_id") or _registration.get_next_id()

    if not user_name:
        return jsonify({"error": "Name is required"}), 400

    if source == "rtsp" and not rtsp_url:
        return jsonify({"error": "RTSP URL is required for RTSP source"}), 400

    camera_source = "local" if source == "local" else rtsp_url

    def on_progress(data):
        socketio.emit("registration_progress", data)

    def on_done(data):
        socketio.emit("registration_done", data)
        if data.get("status") == "success":
            get_pose().refresh_face_database()

    session_id = _registration.start_session(
        source=camera_source,
        user_name=user_name,
        user_age=user_age,
        person_id=person_id,
        on_progress=on_progress,
        on_done=on_done,
    )

    return jsonify({
        "session_id": session_id,
        "status": "started",
        "person_id": person_id,
    })
```

### Điều frontend phải làm đúng
- bấm `New` mới gọi `/registration/start`
- preview modal đọc snapshot backend, không lấy browser webcam làm source of truth

```python
class RecognizerService:
```

```python
    def get_snapshot(self, view: str):
```

## 3.13 Phase 3 Engine — emit snapshot và metric mỗi 5 frame

### Mục tiêu
- đủ mượt cho demo,
- không quá nặng CPU,
- cập nhật log kỹ thuật liên tục,
- Original / Processed lên được.

### Code mẫu callback trong phase 3

```python
def progress_callback(
    clip_stem: str,
    cam_id: str,
    frame_id: int,
    total_frames: int,
    frame_bgr,
    processed_bgr,
    fps: float,
    conf: float,
    adl: str,
    recognizer_service,
):
    from app.api.ws_handlers import emit_metric_log, emit_pose_progress
    import cv2

    # FIX v3: cứ 5 frame mới encode JPEG để nhẹ hơn
    if frame_id % 5 == 0:
        ok1, enc1 = cv2.imencode(".jpg", frame_bgr)
        if ok1:
            recognizer_service.set_original_snapshot(enc1.tobytes())

        ok2, enc2 = cv2.imencode(".jpg", processed_bgr)
        if ok2:
            recognizer_service.set_processed_snapshot(enc2.tobytes())

    emit_pose_progress(
        clip_stem=clip_stem,
        cam_id=cam_id,
        frame_id=frame_id,
        total_frames=total_frames,
        fps=fps,
        conf=conf,
        adl=adl,
    )

    emit_metric_log(
        cam=cam_id,
        fps=round(fps, 2),
        frame=frame_id,
        conf=round(conf, 3) if isinstance(conf, (int, float)) else conf,
        adl=adl,
        event=f"Processing {clip_stem}",
        mode="phase3",
    )
```

### Vì sao đây là tradeoff tốt
- encode JPEG mỗi frame rất nặng,
- 5 frame/lần đủ để người xem cảm giác “đang chạy”,
- CPU/GPU vẫn còn cho pose inference.
