# Prompt cho Codex — sửa CPose UI + backend theo đúng behavior runtime mới

Bạn đang sửa **trực tiếp code hiện có** của dự án CPose.  
Không được refactor lớn, không đổi kiến trúc 3 cột, không thêm framework frontend mới.  
Giữ nguyên stack hiện tại: **Flask + Flask-SocketIO + static HTML/CSS/JS**.

Repo và kiến trúc hiện tại đã có:
- `main.py` chạy Flask app ở `localhost:5000` qua `socketio.run(...)`.
- `app/bootstrap/app_factory.py` serve `static/index.html` tại `/` và static tại `/static`.
- `app/api/routes.py` hiện có các nhóm API `/api/config/*`, `/api/recording/*`, `/api/analysis/*`, `/api/pose/*`, `/api/registration/*`.
- `CPOSE_RULES.md` yêu cầu mọi API phục vụ UI phải nằm trong `app/api/routes.py` và multicam demo phải đi theo luồng tuần tự time-first.  

Các điểm đó đã có trong code/tài liệu hiện tại và phải được tôn trọng khi sửa.  

## MỤC TIÊU CHÍNH

Sửa UI + backend để behavior cuối cùng đúng như sau:

1. **Không auto xử lý video** khi load folder hay load TXT/RTSP.
2. **Load** ở phần Multicam phải là hành động chủ động của user, không auto sync default folder khi vừa mở trang.
3. Nếu user chọn folder video multicam thì chỉ **nạp danh sách clip vào cột VIDEO CLIPS** ở trạng thái chờ (`ready` / `queued`), chưa xử lý ngay.
4. Chỉ khi user nhấn **START ở khối MULTICAM** thì mới bắt đầu:
   - show video gốc ở `ORIGINAL`
   - show video đã xử lý ở `PROCESSED`
   - ghi log kỹ thuật xuống bảng đen phía dưới
   - giữ log cũ, thêm log mới nối tiếp, không xóa log cũ khi sang clip khác
5. Nút **RUN** ở cột `VIDEO CLIPS` **không phải nút xử lý AI**, mà là nút chạy **playlist preview tuần tự** trong `LIVE PREVIEW` cho danh sách clip hiện có.
6. `LIVE PREVIEW` chỉ hiện khi user:
   - nhấn `Preview` trên từng card clip, hoặc
   - nhấn `RUN` để xem tuần tự
   - Không được tự động chiếm `LIVE PREVIEW` khi mới load folder.
7. TXT/RTSP và MULTICAM là **2 luồng mutually exclusive**:
   - Chỉ cho phép **1 luồng chạy tại 1 thời điểm**.
   - Nếu TXT/RTSP đang RUNNING thì khóa Multicam.
   - Nếu Multicam đang RUNNING thì khóa TXT/RTSP.
   - Chỉ khi luồng hiện tại đã STOPPED mới cho phép thao tác luồng còn lại.
8. `ORIGINAL` và `PROCESSED` phải phụ thuộc vào **luồng đang active**. Nếu luồng tắt thì 2 view này không được show.
9. `resources.txt` chứa camera dạng như:
   - `Cam Nha __ rtsp://user:pass@192.168.100.xxx:554/stream2`
   khi load thành công phải populate select ở khung RTSP/MULTICAM để user chọn camera và có thể show preview trong `MULTICAM VIEW`.
10. Face registration hiện chưa hoạt động đúng. Phải sửa để registration dùng backend session thật, có snapshot loop thật, và khi đăng ký xong thì cập nhật đúng lên UI.
11. Lỗi `Range.expand() is deprecated` trong F12 là lỗi từ extension/content script của trình duyệt, **không phải lỗi core của CPose**. Không được chase lỗi đó trong app code.

# A. NGUYÊN TẮC KIẾN TRÚC BẮT BUỘC

## A1. Không auto load default multicam khi mở trang
Nếu hiện tại frontend đang gọi kiểu “sync từ backend” rồi tự nạp `data/multicam` mặc định, hãy bỏ behavior đó.

**Behavior mới đúng:**
- User phải chủ động bấm `LOAD` ở khối MULTICAM.
- Sau đó mới chọn 1 trong 2 cách:
  1. **Load server default multicam folder** (`data/multicam/`), hoặc
  2. **Load local folder từ browser** bằng `webkitdirectory`

## A2. Browser không thể tin cậy absolute Windows path
Ràng buộc kỹ thuật rất quan trọng:
- Frontend/browser không đọc được absolute path thật như `D:\Capstone_Project\data\multicam` theo cách tin cậy do browser sandbox.
- Vì vậy **không được viết JS kiểm tra path string tuyệt đối rồi so sánh**.

### Cách đúng
Hãy triển khai **2 mode load multicam**:

#### Mode 1 — Server default folder
- Nút `LOAD` gọi endpoint backend mới:
  - `POST /api/workspace/load_multicam_default`
- Backend đọc đúng thư mục server-side:
  - `BASE_DIR / data / multicam`
- Backend trả JSON danh sách clip đã parse + sort.
- Frontend render sang `VIDEO CLIPS`.

#### Mode 2 — Local browser folder (optional)
- Nếu muốn vẫn hỗ trợ chọn local folder từ browser:
  - dùng `<input type="file" webkitdirectory multiple>`
  - frontend chỉ đọc các `File` object local
  - render clip queue local
  - không được giả định backend biết absolute path đó

Nếu cần chọn 1 cách duy nhất để tránh phức tạp, **ưu tiên Mode 1: server default folder**.

# B. THIẾT KẾ STATE CHUẨN

Hãy sửa state frontend để có tối thiểu cấu trúc sau:

```js
const state = {
  activeFlow: null, // null | 'rtsp' | 'multicam'
  flows: {
    rtsp: {
      loaded: false,
      running: false,
      selectedCamId: null,
      led: 'off' // off | ready | running
    },
    multicam: {
      loaded: false,
      running: false,
      led: 'off',
      sourceMode: null, // null | 'server_default' | 'local_folder'
      clips: []
    }
  },
  preview: {
    multicamVisible: false,
    originalVisible: false,
    processedVisible: false,
    livePreviewVisible: false,
    activeClipId: null,
    playlistRunning: false,
    playlistIndex: 0
  },
  rtspSources: [],
  eventLogs: [],
  metricLogs: [],
  registration: {
    webcamOpen: false,
    sessionId: null,
    active: false,
    currentUser: null
  }
};
window.state = state;
```

### Quy tắc trạng thái
- `activeFlow === 'rtsp'` khi luồng TXT/RTSP đang chạy.
- `activeFlow === 'multicam'` khi luồng multicam đang chạy.
- Nếu `activeFlow !== null`, toàn bộ nút START bên luồng kia phải bị disable.
- `led` mapping:
  - `off` = đỏ
  - `ready` = vàng/amber hoặc xanh nhạt (đã load nhưng chưa chạy)
  - `running` = xanh lá

# C. HÀNH VI UI CHUẨN

## C1. Khối TXT / RTSP
UI có các phần:
- nút `TXT`
- nút `START`
- nút `STOP`
- select camera RTSP
- đèn trạng thái riêng cho RTSP

### Yêu cầu
1. Bấm `TXT`:
   - mở file picker `.txt`
   - upload lên `/api/config/upload`
   - backend parse camera list
   - frontend gọi `/api/config/cameras`
   - populate select camera
   - `state.flows.rtsp.loaded = true`
   - LED RTSP = `ready`

2. Bấm `START` của RTSP:
   - chỉ được phép nếu:
     - RTSP đã loaded
     - `activeFlow === null`
   - gọi `/api/recording/start` hoặc `/api/pose/start` với `mode: 'rtsp'`
   - `state.activeFlow = 'rtsp'`
   - LED RTSP = `running`
   - khóa toàn bộ control Multicam
   - nếu `ORIGINAL/PROCESSED` đang show thì bắt đầu polling snapshot

3. Bấm `STOP` của RTSP:
   - gọi stop API tương ứng
   - `state.activeFlow = null`
   - RTSP LED = `off`
   - nếu RTSP không còn loaded thì giữ `off`, nếu muốn giữ config loaded thì cho về `ready`
   - `ORIGINAL/PROCESSED` phải clear nếu không có luồng active khác
   - mở khóa control Multicam

### Quy tắc preview RTSP
- Khung RTSP/MULTICAM chỉ show preview camera RTSP khi:
  - RTSP source đã loaded
  - user chọn camera trong select
  - user nhấn nút `eye/show`
- Nếu RTSP flow đã stop thì view RTSP phải ẩn/clear.

## C2. Khối MULTICAM
UI có:
- nút `LOAD`
- nút `START`
- nút `STOP`
- LED riêng cho Multicam

### Behavior mong muốn
1. Bấm `LOAD`:
   - nếu `state.activeFlow === 'rtsp'` và RTSP đang running => **chặn**, hiện toast:
     - `Phải dừng TXT/RTSP trước khi load Multicam`
   - nếu không, mở lựa chọn load nguồn multicam.

### Khuyến nghị UX tốt nhất
Thêm modal hoặc menu nhỏ với 2 lựa chọn:
- `Load server folder (data/multicam)`
- `Load local folder`

#### Lựa chọn A — server default folder
- gọi `POST /api/workspace/load_multicam_default`
- backend parse + sort theo time-first
- trả danh sách clips
- frontend render vào `VIDEO CLIPS`
- set:
  - `state.flows.multicam.loaded = true`
  - `state.flows.multicam.sourceMode = 'server_default'`
  - `state.flows.multicam.led = 'ready'`
- **Không start xử lý**
- `ORIGINAL`, `PROCESSED`, `LIVE PREVIEW` vẫn chưa hiện gì

#### Lựa chọn B — local folder
- dùng `multicamInput` với `webkitdirectory`
- chỉ nạp file video sang queue
- set `ready`
- **không xử lý ngay**

2. Bấm `START` của Multicam:
   - chỉ được phép nếu:
     - `state.flows.multicam.loaded === true`
     - `state.activeFlow === null`
   - gọi API start multicam thật:
     - ưu tiên `POST /api/workspace/start_multicam`
     - hoặc `POST /api/pose/start` với `mode: 'multicam_folder'`
   - set:
     - `state.activeFlow = 'multicam'`
     - `state.flows.multicam.running = true`
     - LED Multicam = `running`
   - khóa toàn bộ control RTSP
   - **chỉ từ lúc này** mới bắt đầu:
     - show `ORIGINAL`
     - show `PROCESSED`
     - đẩy metric logs xuống bảng đen

3. Bấm `STOP` của Multicam:
   - gọi stop API
   - set:
     - `state.flows.multicam.running = false`
     - `state.activeFlow = null`
     - LED Multicam = `off` hoặc `ready` nếu giữ queue
   - `ORIGINAL/PROCESSED` clear nếu không còn luồng active
   - mở khóa control RTSP

### Quy tắc rất quan trọng
- **LOAD chỉ là nạp queue**
- **START mới là xử lý**
- Tuyệt đối không show `ORIGINAL/PROCESSED` chỉ vì đã LOAD.

## C4. VIDEO CLIPS + RUN + LIVE PREVIEW
### Mục tiêu chính
- `VIDEO CLIPS` chỉ là queue / danh sách clip.
- `LIVE PREVIEW` chỉ mở khi user yêu cầu.

### Khi click `Preview` trên card clip
- Không ảnh hưởng AI pipeline
- Chỉ feed video đó vào `LIVE PREVIEW`
- Update label dưới preview

### Khi nhấn `RUN` ở cột VIDEO CLIPS
- Không chạy pose/adl.
- Nó chỉ chạy **playlist preview tuần tự** trong `LIVE PREVIEW`:
  1. mở clip đầu tiên
  2. khi clip kết thúc -> tự động sang clip tiếp theo
  3. tiếp tục tới hết queue

### Ràng buộc
- Nếu AI multicam đang chạy thì `RUN` vẫn chỉ là preview playlist, không được can thiệp vào processing queue.
- Nếu user chọn clip khác trong lúc playlist đang chạy, playlist phải dừng hoặc reset theo UX bạn chọn, nhưng phải nhất quán.

### Code mẫu frontend cho RUN playlist
```js
async function runClipPlaylistPreview() {
  if (!state.flows.multicam.clips.length) {
    toast('Chưa có clip để preview', 'warning');
    return;
  }

  state.preview.playlistRunning = true;
  state.preview.playlistIndex = 0;

  while (state.preview.playlistRunning && state.preview.playlistIndex < state.flows.multicam.clips.length) {
    const clip = state.flows.multicam.clips[state.preview.playlistIndex];
    await playClipInLivePreview(clip);
    state.preview.playlistIndex += 1;
  }

  state.preview.playlistRunning = false;
}

function stopClipPlaylistPreview() {
  state.preview.playlistRunning = false;
}

function playClipInLivePreview(clip) {
  return new Promise((resolve) => {
    const video = document.getElementById('previewVideoLocal');
    const empty = document.getElementById('previewVideoEmpty');
    if (!video) return resolve();

    video.pause();
    video.src = clip.preview_url || clip.url || '';
    empty?.classList.add('hidden');
    video.onended = () => resolve();
    video.onerror = () => resolve();
    video.play().catch(() => resolve());
  });
}
```

## C5. Registration
### Vấn đề hiện tại cần sửa
- UI mở modal được nhưng face registration vẫn chưa chạy end-to-end.
- Phải dùng backend session thật.

### Behavior đúng
1. `Open`:
   - toggle webcam UI thật
   - chỉ mở preview browser-side
   - không tự coi đó là registration session

2. `New` / `Register`:
   - mở flow nhập:
     - name
     - age
     - person_id auto-increment từ backend
   - gọi `/api/registration/next_id`
   - gọi `/api/registration/start`
   - backend trả `session_id`
   - frontend bắt đầu polling:
     - `/api/registration/snapshot/<session_id>`
   - socket nghe:
     - `registration_progress`
     - `registration_done`

3. Khi done:
   - hiển thị bbox + info trên snapshot nếu backend đã vẽ overlay
   - toast success
   - log event

### Code mẫu frontend đăng ký
```js
async function startRegistrationFlow() {
  const name = document.getElementById('regUserName')?.value?.trim();
  const age = document.getElementById('regUserAge')?.value?.trim();
  const personId = document.getElementById('regUserId')?.value?.trim();

  if (!name) return toast('Thiếu tên', 'warning');

  const res = await fetch('/api/registration/start', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      source: 'local',
      name,
      age,
      person_id: personId
    })
  });

  const data = await res.json();
  if (!res.ok) {
    toast(data.error || 'Không thể bắt đầu đăng ký', 'danger');
    return;
  }

  state.registration.sessionId = data.session_id;
  state.registration.active = true;
  startRegistrationPolling(data.session_id);
}
```

```python
@api_bp.route('/workspace/load_multicam_default', methods=['POST'])
def workspace_load_multicam_default():
    video_dir = _app_module.BASE_DIR / 'data' / 'multicam'
    if not video_dir.exists():
        return jsonify({'error': f'Multicam folder not found: {video_dir}'}), 404

    clips = sort_multicam_clips(video_dir.glob('*.mp4'))
    if not clips:
        return jsonify({'error': 'No .mp4 files found in data/multicam'}), 400

    items = []
    for clip in clips:
        cam_id = _app_module.extract_multicam_camera_id(clip) or 'unknown'
        items.append({
            'id': clip.stem,
            'name': clip.name,
            'cam': cam_id,
            'status': 'ready',
            'source_type': 'server_default',
            'server_path': str(clip),
            'preview_url': f"/api/video/{clip.relative_to(_app_module.DATA_DIR).as_posix()}"
        })

    return jsonify({
        'status': 'ok',
        'mode': 'server_default',
        'clips': items,
        'count': len(items)
    })
```

### 2. Start multicam processing
```python
@api_bp.route('/workspace/start_multicam', methods=['POST'])
def workspace_start_multicam():
    body = request.get_json(force=True, silent=True) or {}
    folder = body.get('folder')
    video_dir = _resolve_dir(folder, _app_module.BASE_DIR / 'data' / 'multicam')

    if not video_dir.is_dir():
        return jsonify({'error': f'Directory not found: {video_dir}'}), 400

    recognizer = get_pose()
    clips = sort_multicam_clips(video_dir.glob('*.mp4'))
    if not clips:
        return jsonify({'error': 'No .mp4 files found for multicam'}), 400

    # Không xóa log cũ ở frontend. Backend chỉ enqueue tuần tự.
    for clip in clips:
        cam_id = _app_module.extract_multicam_camera_id(clip) or 'unknown'
        recognizer.enqueue_clip(cam_id, clip)

    return jsonify({
        'status': 'started',
        'mode': 'multicam',
        'total_clips': len(clips)
    })
```

## D4. Recognizer snapshots phải có original + processed riêng
Recognizer service phải giữ 2 snapshot khác nhau:
- original
- processed

Contract hiện tại ở routes đã có:
- `/api/pose/snapshot/original`
- `/api/pose/snapshot/processed`

Codex phải đảm bảo service cập nhật được cả 2 nguồn snapshot này trong lúc processing.

### Code mẫu service-side concept
```python
class RecognizerService:
    def __init__(self, ...):
        self._snapshots = {
            'original': None,
            'processed': None,
        }

    def set_snapshot(self, view: str, frame_bgr):
        ok, buf = cv2.imencode('.jpg', frame_bgr)
        if ok:
            self._snapshots[view] = buf.tobytes()

    def get_snapshot(self, view: str):
        return self._snapshots.get(view)
```

Trong loop phase 3:
```python
if frame_idx % 5 == 0:
    self.set_snapshot('original', original_frame)
    self.set_snapshot('processed', overlay_frame)
```

```js
async function onMulticamLoadClick() {
  if (state.activeFlow === 'rtsp' && state.flows.rtsp.running) {
    toast('Hãy dừng TXT/RTSP trước khi load Multicam', 'warning');
    return;
  }

  const useServerDefault = confirm('OK = dùng folder server data/multicam\nCancel = chọn folder local');

  if (useServerDefault) {
    await loadMulticamServerDefault();
  } else {
    document.getElementById('multicamInput')?.click();

async function loadMulticamServerDefault() {
  const res = await fetch('/api/workspace/load_multicam_default', {
    method: 'POST'
  });
  const data = await res.json();
  if (!res.ok) return toast(data.error || 'Không load được multicam', 'danger');

  state.flows.multicam.loaded = true;
  state.flows.multicam.sourceMode = 'server_default';
  state.flows.multicam.clips = data.clips || [];
  setMulticamLed('ready');
  renderClipQueue();
  pushEventLog('Loaded multicam clips (server)', 'mc');
}
```

# H. ACCEPTANCE CRITERIA

Chỉ coi là xong khi đạt đủ:

1. Bấm `LOAD` ở Multicam không auto xử lý, chỉ nạp clip queue.
2. Nếu dùng server default folder thì clip từ `data/multicam` mới sang `VIDEO CLIPS`.
3. `LIVE PREVIEW` chỉ show khi user bấm `Preview` hoặc `RUN` playlist.
4. `RUN` ở `VIDEO CLIPS` chạy tuần tự preview từng clip.
5. `START` của Multicam mới bắt đầu xử lý AI và mới feed `ORIGINAL/PROCESSED`.
6. TXT/RTSP start => LED RTSP xanh, stop => đỏ hoặc ready theo logic; Multicam cũng tương tự.
7. Chỉ 1 flow được RUNNING tại 1 thời điểm.
8. Nếu RTSP đang chạy thì không load/start Multicam được, và ngược lại.
9. `resources.txt` parse xong phải populate camera select để chọn preview RTSP.
10. Registration chạy session thật qua backend và snapshot thật.
11. Log metric giữ lịch sử xuyên nhiều short video, không bị clear mỗi clip.
12. `ORIGINAL/PROCESSED` clear khi flow stop.
